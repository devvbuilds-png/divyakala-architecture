An aside on scale

Will it hold up with everyone watching at once?

Yes. The app and every video are served from global CDNs — adding viewers just adds edge requests, never a bottleneck. Live-class concurrency is Zoom's job, not ours. The only ceiling worth watching is the database, and at 200–300 users it sits well inside the default tier — liftable later with a single setting, not a rebuild. The system is ready for your expected load and several times beyond it.

§02
How it works · three flows

Three journeys through the system.

i.

Watching a recorded lesson

  1. Student signs in — Supabase Auth verifies them.
  2. The app loads instantly from the Cloudflare edge.
  3. Course data is read from the database; access is checked by security policies.
  4. An Edge Function signs a short-lived video token.
  5. Bunny.net streams adaptive-quality video from the nearest edge.
  6. Progress & submissions are saved back to Supabase.
ii.

Attending a live class

  1. Admin creates a batch session with date, time & Zoom link.
  2. Session is stored in the database against that batch.
  3. Enrolled students see it on their Batch Dashboard — only their batch.
  4. At class time the student clicks through to Zoom.
  5. Zoom hosts the live workshop for up to 100 participants.
iii.

Recording → library

  1. Class opens — Zoom auto-records to the cloud, no button pressed.
  2. Class ends; the recording lands in Zoom Cloud — a 10 GB buffer.
  3. v1: Drdha moves the recording into Bunny from the admin panel.
  4. v2: a Zoom webhook + Edge Function push it to Bunny automatically.
  5. The Zoom Cloud copy is deleted to free the buffer.
  6. It now plays like any lesson — CDN-streamed from Bunny.
